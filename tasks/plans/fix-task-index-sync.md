## Problem (verified on disk 2026-06-03)
`task_index_sync.py` regenerates TASK_INDEX.md from the Convex MC board. It is:
1. **Broken:** shells out to `npx` (to call Convex), but `npx` is not on PATH in the minimal cron/launchd exec environment. Every run dies with `FileNotFoundError: [Errno 2] No such file or directory: 'npx'`. Log `/Users/TJ/.hermes/workspace/tasks/task-index-sync-cron.log` is 2.2MB of identical tracebacks.
2. **Multi-scheduled:** runs from BOTH crontab (`*/30 * * * *`) and launchd (`ai.hitnetwork.task-index-sync`, StartInterval). Likely ~96 failed invocations/day.
3. **Possibly triple-covered:** jobs.json has `sync_task_index_from_convex` (job 582862415f6c, daily 7am, runs `sync-task-index-from-convex.sh`). Reid MUST determine whether this jobs.json job is the canonical/working TASK_INDEX writer that SUPERSEDED the .py — if so, the .py on both crontab+launchd is dead legacy and should be retired entirely, not fixed.

## Fix (only if the .py is the canonical writer)
1. **PATH fix:** the script must call Convex via an absolute binary path, not bare `npx`. Either (a) hardcode the absolute path to `npx`/`bun`/`node` resolved via `which`, or (b) prepend `export PATH="$HOME/.local/bin:$HOME/.bun/bin:/usr/local/bin:/usr/bin:/bin:$PATH"` inside the script's subprocess env, or (c) replace the `npx convex` shell-out with the authed Convex HTTP API call (`tasks:getAll` via the `__serviceToken` pattern, no npx dependency at all — preferred, most robust).
2. **Dedup to ONE scheduler:** keep launchd (more robust on macOS, survives reboots), remove the crontab line. Never both.
3. **Truncate the 2.2MB log** and add log rotation or a `head`-capped write so a future failure can't balloon again.
